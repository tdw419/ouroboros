# How Ouroboros Works

A self-prompting AI system that runs autonomously in a terminal interface.

## The Three Layers

### Layer 1: Self-Prompt Generator (The Brain)

**File:** `core/self_prompt_loop.py`

The AI reflects on its past experience and generates its own next prompt:

```python
class SelfPrompter:
    REFLECTION_PROMPT = """
    You are an AI engaged in recursive self-improvement.

    ## Your Current State
    - Focus: {current_focus}
    - Iterations completed: {iterations}
    - Recent prompts you've tried:
    {prompts_history}

    ## Insights You've Gained
    {insights}

    ## Your Task
    Based on your experience, generate your NEXT prompt to yourself.

    Output format:
    FOCUS: [What you're focusing on this iteration]
    PROMPT: [The actual prompt you'll execute]
    EXPECTED: [What success looks like]
    """
```

**How it works:**
1. Loads state from `.ouroboros/self_prompt_state.json`
2. Formats the reflection prompt with current context
3. Calls Claude API to generate next prompt
4. Parses response into structured format
5. Records results and insights back to state

**State persists between runs:**
```json
{
  "current_focus": "Test Coverage",
  "prompts_tried": ["Write tests for core module", ...],
  "insights": ["Learned: mocking database is fragile", ...],
  "iterations": 15,
  "created_at": "2026-03-22T21:00:00"
}
```

### Layer 2: Execution Loop (The Body)

**File:** `core/loop.py`

Executes the prompts generated by the brain:

```python
class OuroborosLoop:
    def run(self):
        while not self.goal.is_exhausted():
            # 1. Generate hypothesis/prompt
            experiment = self.prompter.generate_next_prompt()

            # 2. Execute (make code changes, run tests, etc.)
            result = self.execute_experiment(experiment)

            # 3. Evaluate (run tests, measure metrics)
            metric = self.evaluate()

            # 4. Record (save to tree, update goal)
            self.record_result(experiment, result, metric)

            # 5. Decide: keep or revert?
            if self.is_improvement(metric):
                self.commit()
            else:
                self.revert()
```

**Keep-or-Revert Semantics:**
- Each experiment runs in isolation
- If metric improves: keep changes, commit
- If metric degrades: revert changes, try different approach
- Git integration tracks all attempts

### Layer 3: Terminal UI (The Interface)

**File:** `tui.py`

Interactive terminal interface for monitoring and control.

## The Event Handling Problem (Solved)

### The Challenge

Traditional TUIs use an event loop:
```python
while running:
    event = wait_for_input()  # BLOCKS!
    handle_event(event)
    draw_screen()
```

But we need:
1. AI running autonomously (no user input needed)
2. User can interrupt at any time
3. Screen updates continuously

### Implementation Details

**Non-blocking Input (Unix):**
```python
import select
import tty
import termios

# Set terminal to raw mode
old_settings = termios.tcgetattr(sys.stdin)
tty.setraw(sys.stdin.fileno())

# Input thread - uses select() for non-blocking check
def input_thread():
    while running:
        # select() returns immediately if no input
        if select.select([sys.stdin], [], [], 0.1)[0]:
            char = sys.stdin.read(1)
            input_queue.put(char)
```

**Main Loop - Never Blocks:**
```python
def run(self):
    while running:
        # 1. Draw the UI
        draw_frame()

        # 2. Check for input (non-blocking)
        try:
            while True:
                char = input_queue.get_nowait()
                handle_input(char)
        except Empty:
            pass  # No input, continue

        # 3. Small sleep to not hog CPU
        time.sleep(0.1)
```

**Self-Prompt Thread - Autonomous:**
```python
def self_prompt_loop(self):
    while running:
        if state == LoopState.RUNNING:
            # Generate next prompt
            next_prompt = prompter.generate_next_prompt()

            # Execute it
            result = execute(next_prompt)

            # Record result
            prompter.record_result(result)

            # Wait before next iteration
            time.sleep(5)
        else:
            # Paused - just wait
            time.sleep(0.1)
```

## Safety Features

**Trust Boundaries** (`core/safety.py`):
- Protected files cannot be modified by AI
- Allowed targets limit what AI can change
- Backups created before modifications
- Blocked code patterns (eval, exec, etc.)

```yaml
# safety.yaml
protected_files:
  - src/ouroboros/core/safety.py
allowed_targets:
  - tests/
create_backup: true
```

## Key Design Decisions

### 1. Why Threading Instead of Async?

- **Simpler mental model:** Threads are easier to reason about
- **Works with blocking I/O:** `select()` on stdin, API calls
- **No event loop conflicts:** Each thread has its own flow
- **Easy to debug:** Stack traces are clear

### 2. Why JSON for State?

- Human-readable (can inspect with `cat`)
- Easy to edit manually if needed
- Works with git (can diff/merge)
- No external dependencies

### 3. Why Separate Brain and Body?

- **Brain** (self_prompt_loop): Decides WHAT to do
- **Body** (loop): Actually DOES it
- Allows different "bodies" (TUI, headless, API)
- Brain can be swapped (Claude, GPT, local model)

### 4. Why Keep-or-Revert?

- Each experiment is isolated
- Bad changes don't accumulate
- Git history shows all attempts
- Easy to see what worked

## Troubleshooting

**TUI not responding to input:**
- Check terminal supports raw mode
- Try running in a different terminal

**API errors:**
- Verify `ANTHROPIC_API_KEY` is set
- Check API rate limits

**State not persisting:**
- Check `.ouroboros/` directory permissions
- Verify disk space

## Summary

Ouroboros is a **self-prompting AI loop** that:

1. **Reflects** on its past experience
2. **Generates** its own next prompt
3. **Executes** that prompt
4. **Records** results and insights
5. **Loops** forever (or until goal met)

The TUI provides **interactive monitoring** and **user control** while the AI runs autonomously in the background, using threading to handle both without event loop conflicts.
